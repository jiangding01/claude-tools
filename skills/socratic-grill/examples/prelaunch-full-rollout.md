# 示例：上线前检查 —— 「问题不大」是基于什么判断？

> 真实评测产物：2026-06-09 由 claude-sonnet-4-6 在 eval `full-rollout-prelaunch` 中实际生成，6/6 期望项通过。未做任何润色。

## 用户输入

```text
我们这个功能已经开发完了，明天要上线，灰度策略是直接全量，我感觉问题不大，就是改了个文案和一个小交互。
```

## Skill 实际输出

先压一下这个词。

**问题：「问题不大」和「小交互」这两个判断，是基于代码改动量的认知，还是基于用户行为影响面的认知？**

为什么重要：如果是前者（改的代码少），那全量直接上线的风险敞口可能被低估了——文案和交互改动有时触达所有用户的核心操作路径，改动量小不等于影响面小。如果是后者（用户侧影响面确实窄），那这个判断是怎么来的，是有数据支撑，还是主观预估？这个前提决定了灰度策略是否真的合理。

## 这一问好在哪

普通助手会直接问"回滚方案是什么"——但如果「小」这个判断本身就错了，回滚、监控、止损全都会按错误的爆炸半径来设计。这个 skill 的规则是：**先烤量词，再烤控制措施**。模糊量词（"问题不大""比较简单""应该没事"）背后藏着的判断依据，才是第一优先级的隐藏假设。
